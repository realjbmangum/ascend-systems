Schema Markup Plan · 2026-05-12

Ascend Systems — Schema Markup Plan

A ship-ready structured-data layer for ascendsystems.ai. Nine schema blocks, all fully populated with real Ascend data, plus the React SPA injection pattern to put them on every page. No fabricated reviews. No placeholders.

TL;DR — ship these five first

  1. Organization — sitewide. Tells Google what the entity "Ascend Systems" is. Foundation for every other schema.
  2. LocalBusiness (ProfessionalService subtype) — sitewide. The single biggest lever for "Charlotte software consultancy" local-pack visibility on a new domain.
  3. Person (Brian Mangum) — sitewide + About page. Anchors the solo-operator trust signal in Google's Knowledge Graph.
  4. WebSite + SearchAction — homepage. Enables sitelinks search box for branded "ascend systems" queries.
  5. FAQPage — homepage (data already exists in FAQSection.tsx). Highest-ROI rich result: 7 Q&A pairs eligible for SERP expansion.

Phase 2 (next 2-4 weeks): five Service blocks, SoftwareApplication + HowTo for the Cost Calculator. Phase 3: Article + BreadcrumbList per case study.

Schema audit — current state

The site is a Vite + React 19 SPA with client-side routing. The existing src/components/SEO.tsx component only injects <title> and <meta name="description"> at runtime via useEffect. There is zero structured data on any page today. view-source: on ascendsystems.ai returns the Vite shell with no application/ld+json blocks, no og: tags, no canonical, no robots directives beyond the default.

This is actually a feature, not a bug. It means we get to design the entire structured data graph from scratch with no legacy to refactor. We can ship a clean, self-consistent @graph object the first time.

Two SPA-specific gotchas to design around

  1. Googlebot does render JS, so useEffect-injected JSON-LD will get picked up — but with a render-budget delay. For the highest-priority sitewide schema (Organization, LocalBusiness, Person, WebSite), we want it in the static index.html head so it's present in the raw HTML response, not after JS hydration.
  2. Per-page schema (FAQ on home, Article on case studies) is fine to inject via the React component because Google re-renders and updates. We just want it written into a single <script type="application/ld+json"> tag with a stable ID we can replace, not duplicated on re-renders.

4. WebSite + SearchAction (homepage)

Why we need it: Enables the sitelinks search box on Google for branded "ascend systems" queries. Also declares the canonical site identity so subordinate pages can reference it via @id.

Where it goes: index.html <head>, sitewide. The SearchAction assumes the site has a /search?q= endpoint — if no real search UI exists yet, drop the potentialAction block until we ship one (Google rejects fake search endpoints).

<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "WebSite",
  "@id": "https://ascendsystems.ai/#website",
  "url": "https://ascendsystems.ai",
  "name": "Ascend Systems",
  "description": "Charlotte technology consultancy — custom SaaS, AI agents, internal tools, legacy modernization, fractional CTO.",
  "publisher": { "@id": "https://ascendsystems.ai/#organization" },
  "inLanguage": "en-US"
}
</script>

Add the SearchAction block only once /search ships:

"potentialAction": {
  "@type": "SearchAction",
  "target": {
    "@type": "EntryPoint",
    "urlTemplate": "https://ascendsystems.ai/search?q={search_term_string}"
  },
  "query-input": "required name=search_term_string"
}

FAQ rich results expand the SERP listing vertically, doubling or tripling click real estate. Pattern B — Per-page <JsonLd> component

For page-specific schema (FAQPage on home, Service × 5 on services, Article on each case study, BreadcrumbList anywhere), use a small reusable React component that injects a <script> tag into the document head and removes it on unmount.

// src/components/JsonLd.tsx
import { useEffect } from 'react';

interface JsonLdProps {
  /** A stable id so we can replace, not duplicate, on re-renders */
  id: string;
  /** Any schema.org-shaped object (or @graph array) */
  data: Record<string, unknown>;
}

export default function JsonLd({ id, data }: JsonLdProps) {
  useEffect(() => {
    const scriptId = `jsonld-${id}`;
    let el = document.getElementById(scriptId) as HTMLScriptElement | null;
    if (!el) {
      el = document.createElement('script');
      el.type = 'application/ld+json';
      el.id = scriptId;
      document.head.appendChild(el);
    }
    el.textContent = JSON.stringify(data);

    return () => {
      const existing = document.getElementById(scriptId);
      if (existing) existing.remove();
    };
  }, [id, data]);

  return null;
}

Used like this on the Home page:

// src/pages/Home.tsx

Pattern C — Optionally extend the existing SEO component

If the team prefers a single SEO surface, extend src/components/SEO.tsx with an optional jsonLd prop:

// src/components/SEO.tsx (extended)
import { useEffect } from 'react';

interface SEOProps {
  title: string;
  description: string;
  jsonLd?: Record<string, unknown> | Record<string, unknown>[];
  jsonLdId?: string;
}

export default function SEO({ title, description, jsonLd, jsonLdId = 'page' }: SEOProps) {
  useEffect(() => {
    document.title = title;

    let meta = document.querySelector('meta[name="description"]');
    if (meta) {
      meta.setAttribute('content', description);
    } else {
      meta = document.createElement('meta');
      meta.setAttribute('name', 'description');
      meta.setAttribute('content', description);
      document.head.appendChild(meta);
    }

    if (jsonLd) {
      const scriptId = `jsonld-${jsonLdId}`;
      let el = document.getElementById(scriptId) as HTMLScriptElement | null;
      if (!el) {
        el = document.createElement('script');
        el.type = 'application/ld+json';
        el.id = scriptId;
        document.head.appendChild(el);
      }
      el.textContent = JSON.stringify(jsonLd);
    }

    return () => {
      const existing = document.getElementById(`jsonld-${jsonLdId}`);
      if (existing) existing.remove();
    };
  }, [title, description, jsonLd, jsonLdId]);

  return null;
}

Validation

Validate every block before and after deploy:

  1. Google Rich Results Test — paste the JSON-LD into search.google.com/test/rich-results, or run it against a deployed URL. Tells you which Google rich-result categories the markup qualifies for. Required to pass with zero errors for Phase 1 entities.
  2. Schema Markup Validator — paste into validator.schema.org for pure schema.org spec compliance (catches things Google ignores but other consumers like AI crawlers care about).
  3. Post-deploy spot check — view-source:https://ascendsystems.ai in the browser, search for application/ld+json, confirm the static block is in the raw HTML response (not just after JS hydration). For per-page schema, open DevTools → Elements → search head for jsonld-.
  4. Search Console enhancements report — once indexed (~1-2 weeks post-deploy), GSC will surface a per-rich-result-type report showing valid/warning/error pages. Monitor weekly.
  5. Curl test — curl -s https://ascendsystems.ai | grep -A 50 'application/ld+json' to confirm static head injection survives the production build pipeline.

Anti-patterns to avoid

  • Don't put schema in dangerouslySetInnerHTML with un-escaped quotes. Always use JSON.stringify() on a JS object — never hand-write JSON-LD as a template literal.
  • Don't duplicate the same @id in multiple places without resolving. If Organization has @id "https://ascendsystems.ai/#organization", every reference (publisher, provider, founder) should use { "@id": "https://ascendsystems.ai/#organization" }, not a full duplicate Organization object.
  • Don't mark up content that isn't visible on the page. Google will flag FAQ schema if the answers aren't rendered. Same for HowTo steps, Service descriptions, etc.
  • Don't ship AggregateRating without real reviews. Manual penalty risk.
  • Don't use placeholder phone numbers in production. The 555 example above is a placeholder — replace or remove before deploy.
  • Don't inject schema in useEffect for top-priority sitewide entities. Use static index.html instead so crawl-budget-constrained bots see it in the first response.
