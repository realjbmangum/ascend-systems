Ascend Systems — Security Review

Full review of the Ascend Systems worker (Cloudflare Workers + D1 + R2), admin CRM, public site, and webhook integrations. Code + live endpoint audit.

TL;DR — Moderate risk · No P0 found

No critical vulnerabilities. The recently-deleted /api/auth/test-login backdoor is confirmed absent from source. No hardcoded secrets, no SQL injection vectors, no unauthorized admin access.

Findings: 15 total — 0 P0, 5 P1, 5 P2, 5 P3.

Most urgent (P1): No rate limiting on any of the four public write endpoints (/api/contact, /api/intake, /api/leads, /api/tools/cost-calculator). An attacker can loop a single IP to flood the leads table and drain the SendGrid quota (50k/month) in under 14 hours. Cloudflare Rate Limiting fixes this in 10 minutes from the dashboard, no code deploy.

P1 — High severity (fix this week)

P1 · 1

No rate limiting on any public write endpoint

Files: worker/src/index.ts:42, :109, :190; worker/src/routes/tools.ts:97

Endpoints: POST /api/contact, POST /api/intake, POST /api/leads, POST /api/tools/cost-calculator

Risk: All four endpoints accept arbitrary JSON, insert into leads, create a task, and optionally trigger a SendGrid email — with no IP-level throttle, no CAPTCHA, and no duplicate-email guard. An attacker can loop /api/tools/cost-calculator from a single IP and flood the leads table and drain SendGrid quota. Each contact/calculator submission also appends a drip enrollment, multiplying email sends. At 1 req/sec this exhausts a day's SendGrid quota in under 14 hours.

// worker/src/index.ts:42 — no rate check before insert
app.post("/api/contact", async (c) => {
  const body = await c.req.json<{ name?: string; email?: string; ... }>();
  if (!body.name || !body.email) { ... }
  const result = await c.env.DB.prepare(
    `INSERT INTO leads (name, email, ...) VALUES (?, ?, ...)`
  ).bind(...).run();
  // No throttle. No duplicate guard. Proceeds to email send.

Fix

Add Cloudflare Rate Limiting rules at the Workers level targeting /api/contact, /api/intake, /api/leads, /api/tools/* by IP: 5 requests/minute per IP. Free tier covers 10k requests. Also add a per-email guard on insert: SELECT COUNT(*) FROM leads WHERE email = ? AND created_at > datetime('now', '-24 hours') — reject if > 0.

P1 · 2

Stripe webhook: HMAC verified but timestamp staleness not checked

File: worker/src/stripe.ts:4–34

Risk: verifyStripeSignature correctly extracts t (timestamp) and v1 (HMAC) from the stripe-signature header and validates the HMAC. It never checks whether the timestamp is within an acceptable window. Stripe's docs specify a 300-second tolerance. Without this check, a captured webhook payload can be replayed indefinitely — an attacker who records a legitimate invoice.paid webhook can replay it to mark invoices paid without actual payment.

// worker/src/stripe.ts:16 — timestamp extracted but never validated for age
if (!timestamp || !hash) return false;
// Missing: const age = Math.abs(Date.now()/1000 - Number(timestamp));
//          if (age > 300) return false;
const key = await crypto.subtle.importKey(...)

Fix

After extracting timestamp: const age = Math.abs(Date.now() / 1000 - parseInt(timestamp, 10)); if (age > 300) return false;

P1 · 3

ADMIN_EMAILS stored as plaintext in wrangler.toml (committed to git)

File: worker/wrangler.toml:7–8

Risk: ADMIN_EMAILS = "bmangum1@gmail.com" is in the [vars] block, which means it is a plaintext Worker variable — not a secret. Committed to GitHub history, visible in the Cloudflare dashboard. This value is the sole factor determining who receives admin-role magic links.

[vars]
APP_ORIGIN = "https://ascendsystems.ai"
ADMIN_EMAILS = "bmangum1@gmail.com"   # <-- plaintext, controls admin auth

Risk rating is P1 not P0 because an attacker still needs control of the inbox to complete auth. But since the email is publicly associated with Brian, the "secret" value is effectively zero.

Fix

Move to a wrangler secret: wrangler secret put ADMIN_EMAILS. Remove from wrangler.toml. The Bindings type already marks it ADMIN_EMAILS?: string, so no code change.

P1 · 4

SameSite=None session cookie without CSRF protection

File: worker/src/routes/auth.ts:96–102

Risk: Session cookie is set with sameSite: "None" + secure: true. SameSite=None was chosen to allow cross-origin requests from the React SPA (different origin from the worker). But this disables all browser-level CSRF protection. Any site can make credentialed cross-origin requests to ascend-api.bmangum1.workers.dev that include the session cookie. CORS limits reading the response — but state-mutating endpoints (PATCH, DELETE, POST) succeed without needing to read it. An attacker can trigger DELETE /api/leads/:id, PATCH /api/leads/:id, etc. via cross-origin POST from any page Brian visits while authenticated.

// worker/src/routes/auth.ts:96–102
setCookie(c, SESSION_COOKIE, sessionToken, {
  httpOnly: true,
  secure: true,
  sameSite: "None",   // <-- disables CSRF protection for cross-origin requests
  path: "/",
  maxAge: SESSION_TTL_DAYS * 24 * 60 * 60,
});

Fix

Either (a) add a custom request header (X-Requested-With: XMLHttpRequest) check on all state-mutating admin endpoints (simple CSRF double-submit), or (b) change to SameSite=Lax and serve the admin SPA from the same registered domain (e.g., move worker to api.ascendsystems.ai). The latter is the correct long-term fix.

P1 · 5

No input length limits on any text field — DoS via oversized payloads

Files: worker/src/index.ts:43–66; worker/src/routes/tools.ts:97–113

Risk: All public endpoints accept body.message, body.process_description, body.name with no length cap. A 100 MB message string will be parsed and inserted into D1. Cloudflare Workers has a 100 MB request body limit, but D1 TEXT fields are unbounded. Attacker can fill the database with junk and degrade read performance.

Fix

Add length validation before insert. Suggested caps: name 255, email 254, message/process_description 5000, company 255.

if (body.message && body.message.length > 5000)
  return c.json({ error: "message too long" }, 400);

P2 — Medium severity (fix this month)

P2 · 6

No security response headers on the Cloudflare Pages site

Impact: No Content-Security-Policy, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, or Permissions-Policy. No public/_headers file exists. The admin SPA can be iframed by any origin (clickjacking). Without CSP, any XSS that does exist has no secondary containment.

Fix

Create public/_headers (Cloudflare Pages picks this up automatically):

/*
  X-Frame-Options: DENY
  X-Content-Type-Options: nosniff
  Referrer-Policy: strict-origin-when-cross-origin
  Permissions-Policy: camera=(), microphone=(), geolocation=()
  Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src https://fonts.gstatic.com; connect-src 'self' https://ascend-api.bmangum1.workers.dev; img-src 'self' data: https:; frame-ancestors 'none'
P2 · 7

File downloads from R2 serve attacker-controlled Content-Type

File: worker/src/index.ts:581, 587–595

Risk: File downloads set Content-Type from obj.httpMetadata?.contentType, which was stored at upload from file.type || "application/octet-stream" — i.e., from the browser, never validated server-side. An admin who uploads an HTML file would have it served as text/html — browsers render rather than download, enabling stored XSS via the file download URL.

Fix

Server-side, override stored contentType on download to application/octet-stream. If preview is needed, maintain a strict allowlist (image/jpeg, image/png, application/pdf).

P2 · 8

process_description stored verbatim without length limit

Files: worker/src/routes/tools.ts:134; worker/src/email.ts:162–165

Risk: Free-text from the cost calculator gets inserted into leads.message and embedded into the sendAdminAlert HTML email. escapeHtml() is correctly applied (no XSS). But no length limit — a 1 MB description causes a large email + DB row.

Fix

Cap process_description at 2000 characters (same fix as Finding 5).

P2 · 9

CORS allows localhost:5173 in production worker

File: worker/src/index.ts:29

Risk: http://localhost:5173 is in the production CORS allowlist. Any browser tab running a local server on port 5173 can make credentialed cross-origin requests to the live worker. Low-probability but a developer machine with a malicious local process could authenticate as admin against production.

Fix

Remove http://localhost:5173 from the production CORS list. Use wrangler dev --local for local dev, or a separate wrangler.dev.toml.

P2 · 10

Stripe signature comparison uses string equality (not constant-time)

File: worker/src/stripe.ts:33

Risk: return computed === hash short-circuits on first differing byte. Remote timing attacks against HMAC over the internet are extremely difficult, but best practice is constant-time compare.

Fix

Length-check first, then crypto.subtle.timingSafeEqual on equal-length buffers.
